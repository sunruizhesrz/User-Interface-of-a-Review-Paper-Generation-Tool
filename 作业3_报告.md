## 二、软件模块设计报告

### 2.1 设计方法

成员 F 的核心定位是系统的"门面"与"粘合剂"。设计思路分为两个维度：

**界面层（app.py）**：采用 Gradio Blocks API 构建响应式 Web 界面，通过事件绑定将用户操作映射到后端流程。选择 Gradio 而非 Streamlit 或纯 Flask+HTML 的原因在于：Gradio 对 LLM 应用的进度流（`gr.Progress`）、Markdown 渲染、文件下载等有原生支持，与 LangGraph 的流式输出契合度高，且开发速度快，适合 Sprint 1 快速验证。

**集成层（pipeline.py）**：采用"防御性导入 + Mock 兜底"模式。通过 `try-import` 机制，在各成员模块未就绪时自动降级为 Mock 函数，确保流程骨架可独立运行、可演示，且对其他成员的开发进度零阻塞。

**导出层（exporter.py）**：采用策略模式，针对三种导出格式（Markdown / PDF / Word）分别实现导出函数，并通过统一的 `export()` 接口对外暴露，调用方无需感知内部实现细节。PDF 导出实现了 pandoc → reportlab 的双重 fallback，提升了环境适应性。

### 2.3 技术原理

#### Gradio Blocks 界面架构

```
gr.Blocks
├── gr.HTML（标题）
├── gr.Row
│   ├── gr.Column（输入区）
│   │   ├── gr.Textbox（主题输入）
│   │   ├── gr.Slider（年份 / 文献数量）
│   │   ├── gr.Dropdown（语言选择）
│   │   └── gr.Button（触发按钮）
│   └── gr.Column（输出区）
│       ├── gr.Textbox（执行日志）
│       ├── gr.Markdown（内容预览）
│       └── gr.Row（下载按钮组）
└── gr.Examples（快速示例）
```

Gradio 的事件系统基于 WebSocket 长连接，`gr.Progress` 通过服务端向客户端推送进度更新，实现实时反馈效果，无需前端轮询。

#### 主流程串联逻辑（pipeline.py）

```
run_pipeline(topic, year_from, max_papers, language)
    │
    ├── [Step 1] search_arxiv()         # 成员 C：检索论文元数据
    ├── [Step 2] download_pdf()         # 成员 C：下载 PDF 全文
    ├── [Step 3] parse_pdf()            # 成员 D：PDF 文本提取
    ├── [Step 4] embed_chunks()         # 成员 D：文本向量化
    ├── [Step 5] add_documents()        # 成员 B：写入 ChromaDB
    ├── [Step 6] run_survey_generation()# 成员 A：LangGraph 编排
    │               │
    │               ├── retrieve()      # 成员 B：RAG 检索
    │               └── generate_section() # 成员 E：章节生成
    └── 整合 Markdown → 输出 dict
```

每步均有 `on_progress` 回调，实时向界面推送状态文字，确保用户感知到系统的执行进度。

#### 防御性导入模式（关键设计决策）

```python
try:
    from graph import run_survey_generation as _real_run
    HAS_A = True
except ImportError:
    HAS_A = False
    def _real_run(user_input: str) -> dict:
        # Mock 实现，返回占位数据
        ...
```

这一设计使得成员 F 的模块在 Sprint 1 中即可独立运行和演示，不受其他成员进度影响，同时在正式模块就绪后只需删除 `try-except` 分支即可无缝切换。

```python
def export(content: str, fmt: str = "md", output_dir: str = "./output",
           filename: str = "survey") -> str:
    fmt = fmt.lower().strip(".")
    output_path = os.path.join(output_dir, f"{filename}.{fmt}")
    dispatch = {
        "md":   export_markdown,
        "pdf":  export_pdf,
        "docx": export_docx,
    }
    return dispatch[fmt](content, output_path)
```

## 三、风险分析与项目风险矩阵

### 3.1 风险识别

对本项目进行全面的技术风险、商业风险和项目风险识别，共梳理出以下 10 项优先级最高的风险：

#### 技术风险

**R1 - LLM 幻觉与引用失真**：大模型在生成综述时可能捏造不存在的文献引用，导致学术不诚信问题。这是综述生成工具的核心技术挑战，可能严重损害系统可信度。

**R2 - RAG 检索质量不足**：若向量检索召回的段落与用户主题相关性较低，则生成的综述内容会出现跑题或浅薄问题，直接影响产品价值。

**R3 - PDF 解析失败率高**：部分论文为扫描版 PDF 或带有复杂排版（双栏、公式密集），PyMuPDF 可能无法正确提取文本，导致知识库数据质量下降。

**R4 - LLM API 限速与费用超支**：调用 OpenAI/Anthropic 等商用 LLM API 时，高并发场景下可能触发限速（Rate Limit），且 API 费用随使用量线性增长，可能超出项目预算。

**R5 - 系统性能瓶颈**：对 30+ 篇论文进行端到端处理（下载、解析、向量化、生成）预计耗时 10-30 分钟，用户等待体验差，可能导致超时或用户流失。

#### 商业风险

**R6 - 版权与合规风险**：自动下载和处理受版权保护的 PDF 论文（如 IEEE、Elsevier 付费期刊）可能面临法律风险，部分学术出版商对 AI 内容挖掘有明确限制条款。

**R7 - 目标用户需求偏差**：实际研究人员对综述质量的要求极高（需覆盖全面、分析深入），当前 RAG + LLM 生成方案在特定领域的综述深度可能无法满足专业研究者需求，导致产品定位偏移。

**R8 - 竞品压力**：Elicit、Semantic Scholar、ResearchRabbit 等成熟学术工具已具备文献综述辅助功能，差异化竞争壁垒尚不清晰，可能影响产品市场接受度。

#### 项目风险

**R9 - 团队模块接口不兼容**：六个成员各自独立开发，在 Sprint 1 结束时进行代码合并时，函数签名、数据格式（如 Chunk 结构、State 字段）可能存在不一致，导致集成耗时大幅超出预期。

**R10 - 关键依赖库版本冲突**：项目依赖 LangGraph、ChromaDB、Gradio、PyMuPDF 等多个第三方库，不同操作系统（Windows/macOS/Linux）下的依赖版本兼容性问题可能导致环境搭建困难。

## 五、个人总结与反思

### 贡献与参与度

本次 Sprint 1，我全程参与了团队需求讨论会议（线下），主动认领了系统集成这一横向职责。在开发过程中，除完成自身三项 Sprint 1 任务外，还通过定义 Mock 接口标准，帮助其他成员明确了模块间的协作边界，减少了联调阶段的不确定性。
